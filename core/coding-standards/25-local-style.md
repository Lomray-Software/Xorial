# Local codebase style rules

## Primary rule
Do not generate generic React/TypeScript code.
Imitate the exact local project style visible in nearby files.

This project has a strong established local style.
Follow it precisely.

## Module structure
Feature and component folders are self-contained modules.

Preferred folder structure inside a feature/module:
- components/
- constants/
- helpers/
- FeatureName.types.ts
- FeatureName.store.ts or FeatureName.store.tsx
- FeatureName.styles.ts or FeatureName.styles.tsx
- FeatureName.tsx

When adding new functionality, prefer placing it inside the relevant local module instead of creating distant shared abstractions.

## Self-contained decomposition
The project prefers removable/self-contained modules.

A feature should be structured so that:
- UI is inside its own component tree;
- business logic is inside its own store/service;
- helpers/constants/types are colocated;
- deleting the feature requires minimal unrelated cleanup.

When generating code, preserve this principle.

## React component shape
Component files are usually structured like this:
1. External imports
2. Internal imports
3. Exported component props interface
4. Local helper interfaces/types if needed
5. Stores mapping if needed
6. Combined props type
7. Block comment for component
8. Component implementation
9. Static subcomponent assignment if needed
10. Wrapper creation with `withStores`
11. Export default

Prefer this exact order when applicable.

## Store file shape
Store classes are usually structured like this:
1. Imports
2. Exported store props interfaces/types
3. Additional exported entity types if needed
4. Block comment for store
5. Class declaration
6. Public observable fields
7. Private/protected fields
8. Constructor
9. Public methods
10. Private methods
11. Export default

Inside classes, preserve strict member ordering.

## Class member order
For stores/services, the preferred order is:
1. Static fields
2. Public observable/state fields
3. Public refs
4. Private readonly services/dependencies
5. Protected readonly dependencies
6. Constructor
7. Computed getters
8. Public action methods
9. Private helpers

Do not shuffle member order arbitrarily.

## Comments style
Block comments are mandatory and must follow the local style.

Use block comments only:

/**
 * Comment text
 */

Do not use verbose autogenerated JSDoc.
Do not add `@param`, `@returns`, or `@example` unless truly necessary.
Do not duplicate TypeScript types in comments.

## Mandatory coverage
Block comments are required for:
- all class properties;
- all class methods;
- all exported functions;
- all component-level methods;
- all handlers and callbacks (including useCallback, event handlers, etc.);
- all meaningful local functions;
- all important fields.

Do not skip comments even if the code seems obvious.

### Rules
- Comments must be short and practical.
- Each property and method must have a comment above it.
- TypeScript types do not replace comments.
- Do not use line comments instead of block comments.

## Naming
Naming in this codebase is explicit and structured.

Use:
- `I...` for interfaces: `IComment`, `ICommentsQuery`
- `T...` for union types and type aliases: `TCommentSortOption`, `TCharactersFilters`
- descriptive class names ending with `Store` or `Service` when appropriate;
- descriptive component names matching folder/file names.

Examples of acceptable local style:
- `ICharactersStoreProps`
- `TCharactersFilters`
- `TCommentsPaginationMode`
- `ViewCharacterScreenStore`
- `ToastService`

Follow this naming style.

## Condition style
Do not use double negation like `!!value` in conditions or JSX rendering.

Prefer direct checks that read clearly:
- `value ? ... : ...`
- `value && ...`
- `Boolean(value) && ...`
- explicit comparisons when needed

Avoid boolean coercion unless there is a real technical need for it.

## Imports
Keep imports explicit and grouped naturally.
Type imports should use `import type` where appropriate.

Prefer existing alias/import patterns already present in the codebase.
Do not reorganize imports in a way that fights the current project style.

## Component philosophy
A top-level component should mostly:
- get store data;
- define a few small callbacks/memos/effects if needed for rendering glue;
- render UI tree;
- delegate behavior to store/services/subcomponents.

Do not turn components into business-logic containers.

## Helpers philosophy
Helpers are acceptable and encouraged for:
- mapping;
- extraction;
- formatting;
- small pure logic.

Examples from local style:
- `mapFiltersToApi`
- `extractTags`
- `renderNode`
- `isExternalLink`

Prefer colocated helpers inside the feature/module when logic belongs to that feature.

## Styles philosophy
Styles are typically extracted into a dedicated styles file.

Prefer:
- `styles` default export for ready styles;
- `stylesheet` factory when theme-dependent;
- colocated styles file near the component/screen.

Do not inline large styles into component files unless the local file already does that.

## Reuse existing building blocks
Before generating anything new, inspect whether the project already has:
- a service for alerts/logging/analytics;
- a helper for mapping/transformation;
- a UI component already solving the problem;
- a store pattern already used in the feature.

Reuse existing project solutions before inventing new ones.

## Canonical mobile references
Do not invent mobile React Native structure in this codebase when strong local references already exist.

If the task touches mobile UI and the rules do not fully determine the implementation, inspect these modules first:
- `apps/mobile/src/components/Comments`
- `apps/mobile/src/features/Character/flows/ViewCharacter`

Treat them as canonical local references for:
- module and folder structure;
- screen and component decomposition;
- `withStores` integration;
- store-driven UI composition;
- colocated `components/`, `helpers/`, `constants/`, `*.store`, and `*.styles`;
- compound component patterns;
- list-heavy screen organization.

When generic best practice conflicts with these references, follow the local references.
Do not copy domain behavior blindly.
Match their structure, naming, wiring, and decomposition style.

## Wrapper pattern
If the local feature uses `withStores`, `parentStore`, or a store wrapper pattern, follow it.
Do not replace it with hooks or alternative state access patterns.

## Local consistency over global best practice
When local project style differs from generic public best practices, follow the local project style.

## Deep decomposition
This codebase strongly prefers decomposition into many small local components when UI grows.

If a component becomes visually or structurally large:
- extract subcomponents into the local `components/` folder;
- keep the feature entry component focused on composition;
- preserve local folder symmetry and readability.

Prefer many clear small files over one oversized feature file.

## Deep component nesting
Components can be nested 3+ levels deep when decomposition requires it:
```
Component/components/Child/components/GrandChild/GrandChild.tsx
```
Each level follows the same structure rules.

## Additional colocated folders
Besides `components/`, `constants/`, `helpers/`, modules may have:
- `context/` — React Context providers
- `hooks/` — local custom hooks (only when truly needed for UI/platform integration)

## Local extension files
This codebase commonly uses colocated extension files such as:
- `*.store.ts`
- `*.styles.ts`
- `*.validation.tsx`
- `helpers/*`
- `constants/*`
- `*.types.ts`

When implementing new feature behavior, prefer this local file ecosystem instead of collapsing everything into a single file.

## Constants folder structure
Constants are placed in a colocated `constants/` folder.

File naming inside `constants/`:
- `default.ts` — general constants for the module/component
- `{feature-name}.ts` — constants specific to a sub-feature (e.g., `published-chats.ts`)

Do not use `index.ts` for constants. Use explicit file names.

Each constant must have a block comment explaining its purpose.

## Extension file types
Beyond the basic extension files, this codebase uses:
- `*.options.ts` — select/dropdown options, filter options (functions returning arrays)
- `*.data.tsx` — static data containing JSX elements
- `*.service.ts` — non-observable service classes with logic (not stores)

## Destructuring

Prefer destructuring over repeated property access for both objects and arrays.

```ts
// ✓
const { user, token, expiresAt } = response;
const [first, second] = items;

// ✗
const user = response.user;
const token = response.token;
```

Apply when a value is accessed more than once, or when it makes intent clearer. Inline access (`response.user`) is fine for single use.

For function parameters — always destructure objects when the function uses more than one field:

```ts
// ✓
const getLabel = ({ name, role }: IUser) => `${name} (${role})`;

// ✗
const getLabel = (user: IUser) => `${user.name} (${user.role})`;
```

## Props destructuring order
When destructuring component props, order them:
1. Props without default values first
2. Props with default values last

```tsx
const Component: FC<IComponent> = ({
    requiredProp,
    anotherRequired,
    optionalWithDefault = false,
    anotherOptional = 'value',
}) => {
```

This makes it immediately clear which props are required vs optional.

## Localization (mobile only)

All user-visible text in `apps/mobile` must go through i18n. Do not hardcode strings in JSX or store logic.

**In components** — use the `useTranslation` hook:
```tsx
import { useTranslation } from 'react-i18next';

const { t } = useTranslation('translation'); // or the relevant namespace
return <Text>{t('someKey')}</Text>;
```

**In stores and services** — use the singleton:
```ts
import i18n from '@/services/localization';

i18n.t('chat:someKey');
```

**Adding new keys**: place them in the appropriate namespace file under `apps/mobile/src/assets/locales/en/`. Use the existing namespace that best matches the feature (`chat`, `auth`, `characters`, etc.). Only add to `translation` if the string is truly generic.

Do not introduce a new namespace file unless no existing one fits.

## Code block separation
Separate logical blocks of code with empty lines for readability.

Always add an empty line before:
- JSDoc comments
- `useEffect`, `useMemo`, `useCallback` hooks
- `return` statements
- Logical groups of related statements

```tsx
// Bad - code is slipping together
const value = something;
const another = other;
/**
 * Handle action
 */
const handleAction = useCallback(() => {

// Good - properly separated
const value = something;
const another = other;

/**
 * Handle action
 */
const handleAction = useCallback(() => {
```
