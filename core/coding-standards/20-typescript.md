# TypeScript rules

## TypeScript-first
This codebase is TypeScript-first.
Use TypeScript as the primary source of correctness and intent.

## Typing principles
- Prefer explicit, readable types.
- Avoid unnecessary generic complexity.
- Avoid type gymnastics unless they provide real value.
- Keep types maintainable and debuggable.
- Use interfaces/types in the style already established in the local code.

## JSDoc and comments
Do not replace useful block comments with verbose JSDoc boilerplate.
Use comments for intent, not for repeating type information.

Preferred style:

/**
 * Loads user profile data and synchronizes the store state.
 */

Avoid large autogenerated docblocks unless explicitly requested.

## Naming
Use names that are:
- clear;
- moderately concise;
- aligned with surrounding code.

Do not aggressively shorten names.
Abbreviations are acceptable only when they are obvious and common in context.

## Functions
Prefer small functions.
A function should ideally have one responsibility.

If a function grows too much:
- split it;
- move parts into helpers;
- move behavior into a store/service if it is business logic.

## Error handling
Exceptions are allowed.
Do not silently swallow errors.

Prefer the existing project error-handling style.
If the project uses dedicated logging or reporting services, use them instead of console logging.

## Logging
Avoid console.log in production-style code when the project already has logging services or infrastructure.
Reuse existing logging/reporting tools.

## Imports and dependencies
- Reuse existing dependencies and utilities first.
- Do not introduce a new package if the project already has an internal solution.
- Keep imports tidy and aligned with local conventions.

## Code generation constraints
When generating code:
- do not create overly abstract helper types;
- do not introduce unnecessary wrappers;
- do not generate “enterprise” boilerplate for simple tasks;
- do not create interfaces without clear need.
